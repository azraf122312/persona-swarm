01 / the problem

One bot finds one kind of bug.

A scripted happy-path test only ever fails the way it was written to. Your users aren't one person — they're impatient, cautious, on a phone, using a screen reader, distrustful, distracted. Each one gets stuck somewhere different, and a single bot never sees it.

one bot
  • follows the path it was scripted for
  • reports the bug it was told to look for
  • never feels friction — it has no patience to lose
a persona swarm
  • eight users reason and behave differently
  • each abandons, hesitates, and rage-clicks on its own
  • divergent coverage one bot could never produce
02 / pipeline

From a URL to a verdict in five steps.

  1. 01

    Map the site

    A crawler walks the target once and builds a page graph the swarm uses for orientation.

  2. 02

    Spawn the swarm

    One agent per persona launches — each in an isolated browser, with its own viewport and mindset.

  3. 03

    Navigate in character

    Every agent perceives the live page and decides its next move as that persona, logging friction.

  4. 04

    Judge the runs

    An independent judge scores each persona, finds shared blockers, and ranks the fixes.

  5. 05

    Diff for regressions

    Every run is remembered. Re-run later — the swarm flags any persona that went from success to failure since last time.

06 / honest questions

What people ask before pasting a URL.

Real. Each persona is an LLM driving its own headless Chromium browser through Playwright. Every persona has a distinct system prompt — its patience, its viewport, the things that frustrate it — and it genuinely perceives the live page and clicks through your app. The console on this page is a scripted demo; a real run drives real browsers.

Persona Swarm runs locally — there is no Persona Swarm cloud. The target URL, your goal, and your API key go only to the server you run yourself (server.py). Your key is held in memory for the run and never written to disk. Reports are saved to your own reports/ folder.

Yes. The personas need an LLM to reason and act, so you bring an Anthropic or OpenAI key. The run calls your provider directly — you pay them for the tokens, and there's nothing to pay us.

A Playwright test confirms the path you scripted still works. Persona Swarm finds the paths you never scripted. The personas wander off the happy path on purpose — an impatient power user, a screen-reader user, a skeptical shopper — and each one gets stuck somewhere your test suite has no reason to look.

It reads, scrolls, types, and clicks — the run is exploratory. The personas are looking for friction, not trying to fire destructive actions. If your goal touches anything irreversible, point the swarm at a staging URL and re-run against production once it's clean.

Yes — that's a primary use case. The report names each friction point, the persona who hit it, and ranks the fixes by priority, so you can hand a clear, prioritized list straight to the LLM writing your code.

On top of the persona UX runs, every swarm also runs a static technical audit of the mapped site. It catches: broken links (HEAD-tested), SEO & meta gaps (missing title, description, h1, canonical, Open Graph), accessibility (images without alt, form inputs without labels), copywriting bugs (lorem ipsum live in prod, TODO / TBD placeholders, vague "click here" link text), UI edges (tap targets under 32×32px), mixed content (http resources on https pages), and auth softlock signals (login forms with no visible error region). Audit findings get merged with persona findings into one ranked fix list.

A few minutes. The site is mapped once, then up to eight persona browsers run in parallel. More personas, more steps, and more pages cost more time and more tokens — all of which you control in Setup.

Yes. Every run is saved, keyed to its URL and goal. Re-run later and the swarm diffs against the previous run — a persona that slipped from success to failure, a new blocker, a dropped health score — and flags it at the top of the report.
